章节 0 · 全景

一次点"运行"发生了什么

整个系统是 Go 当 Web 壳 + Python 当 AI 引擎。Go 跑 HTTP，Python 用 langchain/langgraph 算，两头用 stdin/stdout 的 JSONL 通信。

调用时序

浏览器 ──POST /api/affiliate/run──▶ Go Handler
                                     │
                                     ├─ Validate()
                                     │
                                     ├─ app.RunAffiliateFlow()
                                     │     └─ worker.Call(JSONL 请求)
                                     │              │
                                     │              ▼ stdin
                                     │         Python worker (main.py)
                                     │              │
                                     │              ├─ bridge.runtime.dispatch()
                                     │              └─ graph.run_affiliate_graph()
                                     │                     normalize → fetch → commission
                                     │                       → retrieve → generate → finalize
                                     │              ▲ stdout
                                     ◀── JSONL 响应 ─┘

关键文件对照

文件职责
go/cmd/studio/main.goGo 进程入口，启动 worker + ListenAndServe
go/internal/httpserver/路由 + Handler + JSON I/O
go/internal/worker/client.go管理 Python 子进程，发 JSONL
python/main.pyPython worker，按行读请求、写响应
python/bridge/runtime.py装配 Tool / Retriever / Model，action 分发
python/graph/affiliate_graph.py整个业务流的状态机核心

为什么这么拆

  • Go 负责 I/O 与编排：HTTP、静态资源、超时控制、健康检查这些 Go 的长项。
  • Python 负责 AI 推理：因为 LangChain / LangGraph / transformers 生态都在 Python。
  • 子进程 + stdin/stdout：既不用搭 HTTP server，也能被 Go 监管生命周期，错了就拉起来。

动手练习

  1. 打开 Studio，点一次"运行完整流程"。
  2. 看 Go 的 log，再看返回 JSON 里的 trace，把每个节点和上面的时序图对上号。
  3. 把 python/main.py 的 print 替换成打印到 stderr，看 Go worker 里是怎么被丢掉/保留的（见 client.go 里的 io.Copy(io.Discard, stderr)）。
章节 1 · Prompt 工程

ChatPromptTemplate：把"字符串拼接"升级成可复用的模板

一句话：ChatPromptTemplate 就是带占位符的多角色模板，format_messages 填参数后给 LLM。

对应代码

python/labs/prompt_lab.py

为什么这么写

  • 角色分工：system 立人设、立规则；human 放动态输入。这是 chat 模型的通用约定。
  • 变量外置：所有 {platform} / {title} 都通过参数注入，能被 LangGraph 的节点、或 prompt lab 的调试页复用。
  • 可观测：最后用 format_messages 渲染出来的文本会回传前端（debug.rendered_prompt），方便人眼对照。

背后的 AI 概念

Prompt 工程的核心不是"写一段话"，而是 把动态输入和固定约束解耦。模板化后，你可以 A/B 两套 system prompt、沉淀多语言版本、做 few-shot（把示例放在 messages 列表里）。LangChain 的 ChatPromptTemplate 就是这套思路的骨架。

动手练习

  1. 把 system 里的"affiliate copywriting assistant"改成"严谨的电商合规审核员"，再跑一次，看 result.copy.risk_notes 有没有变化（注意：当前 mock 模型不会真的响应，但渲染的 prompt 会变——这就是分层调试的意义）。
  2. 给 prompt 加一个 few-shot 示例，试试在 from_messages 列表里加 ("human", "...")、("ai", "...")。
章节 2 · 结构化输出

Pydantic Schema：让 LLM 吐 JSON 不再玄学

一句话：用 Pydantic 定义"我要什么结构"，下游就能稳定消费。

对应代码

python/schemas/models.py

为什么这么写

  • Schema 先行：API 层、前端、测试都依赖同一份契约，省得 LLM 时不时少字段崩掉。
  • Literal 约束枚举：decision 只能是三选一，这种"强约束"对业务路由很重要（见第 5 章）。
  • alias/serialization_alias：业务想用 copy 这个词，但 Pydantic 基类保留了这个属性名，所以内部叫 copy_output，对外 JSON 照样叫 copy。

背后的 AI 概念

Structured Output 是让 LLM 从"写文章"转向"填表单"。工程上通常两种做法：(a) 函数调用 / Tool schema（OpenAI、Anthropic 的 tool use）；(b) 引导 LLM 输出 JSON，再用 Pydantic 校验。我们这里用 (b)，更贴近本地模型场景。

动手练习

  1. 给 AffiliateCopy 加一个 call_to_action: str 字段，看看 mock 模型的生成是否还能通过校验。
  2. 在 LocalModelAdapter 里故意返回少一个字段的 JSON，观察 Pydantic 抛啥异常、最后怎么被 dispatch 里的 except Exception 包成 generation_error。
章节 3 · RAG 检索

检索增强：索引 / 召回 / 压缩 三步

一句话：RAG 不神秘，就是把外部知识塞进 prompt 之前那几步预处理。

对应代码

  • python/rag/indexer.py — 加载知识库文档
  • python/rag/retriever.py — 关键词召回 + 句级压缩
  • assets/kb/ — 我们的"知识库"（markdown 文件）

为什么这么写

  • Mock 先跑通，再升级：用关键词交集模拟"相似度打分"，不依赖 embedding 模型，下单机零配置就能演示完整链路。
  • search + compress 两阶段：真实 RAG 里你也会先召回（粗筛），再做 Contextual Compression（只留相关句），减少塞给 LLM 的 token。

背后的 AI 概念

  1. Indexing（indexer.py）：把原文切块、存到向量库。
  2. Retrieval（search）：用查询的向量在向量库里找 Top-K。真实版用 Embedding 模型 + FAISS/Chroma。
  3. Contextual Compression（compress）：只抽出每个文档里跟查询最相关的片段，节省上下文空间。

动手练习

  1. 在 assets/kb/ 里加一份新 markdown，不改代码再跑一次，验证 indexer 自动发现（详见 indexer.py）。
  2. 把 KeywordRetriever 换成基于 sentence-transformers 的向量检索：保持接口 (search / compress) 不变，bridge/runtime.py 换实现即可。这是典型的"适配器模式"练手。
章节 4 · Tools

工具调用：让 LLM 不必"硬编"世界的事实

一句话：Tool 就是 LLM 拿不到/记不住的数据，交给后端确定性查一下。

对应代码

  • python/tools/product_tools.py — 商品信息查 fixtures
  • python/tools/commission_tools.py — 佣金率查 fixtures

为什么这么写

  • 默认兜底，严格可选：默认未命中返回带 fallback: True 标记的占位 fixture（id = fallback-unknown），让 Graph 能跑通展示全链路；真要生产级"宁可失败也不让 LLM 胡编商品"，构造时传 strict=True 即恢复抛错行为。
  • 输入兜底：URL 优先，其次关键词模糊匹配。真实项目里这里就是"调用电商商品 API"。
  • 错误可观测：兜底会打 WARNING；严格模式下 bridge/runtime.py 把 LookupError 翻译成带 action/session_id 的 tool_error，同时写进 worker stderr（Go 侧以 [worker] 前缀转发），排查不再只有一句话。

背后的 AI 概念

Tool Use / Function Calling：让 LLM 以"调用函数"形式拿到事实（价格、库存、用户画像）。我们这里是"Graph 层调度工具"，即 Graph 里某个节点主动调 tool，再把结果喂给下一个节点的 prompt；另一种是让 LLM 自主决定调用哪个工具（ReAct / OpenAI tools），第 5 章会提到。

动手练习

  1. 加一个 inventory_tool，在 commission 通过之后调用，把库存 < 10 的商品也归入 rejected。涉及的改动点：新建 tool → runtime.py 装配 → affiliate_graph.py 加节点和边。
章节 5 · LangGraph

StateGraph：把业务流从 if/else 搬到"图"里

一句话：节点是一个个 Python 函数，边描述节点之间的跳转，状态沿着边流动。

对应代码

python/graph/affiliate_graph.py

为什么这么写

  • TypedDict 做状态：每个节点接收 state、返回增量 dict，LangGraph 自动合并。比纯 dict 多了类型提示，比 class 少了样板。
  • 条件边（add_conditional_edges）让业务规则显式：佣金没过 → 直接 finalize_rejected，过了 → 走检索 + 生成。
  • Trace 就是状态的副产品：每个节点都往 state["trace"] 里追加，前端拿到整条链路，天然可观测。

背后的 AI 概念

Agent = 带决策的状态机。LangGraph 的好处是"你画什么图就执行什么"，不像纯 LLM Agent 那样靠 prompt "骗"模型规划。本质上这是 确定性业务流 和 LLM 生成 的混合：流程由代码保证正确，生成部分由 LLM 负责创造性。

动手练习

  1. 加一个节点 translate_copy，接在 generate_structured_copy 之后，把生成的 copy.title 翻译到 locale。
  2. 把 route_commission 升级成"三叉"：rejected / needs_review / accepted，needs_review 分支走一个新节点 request_manual_review。
章节 6 · Go ↔ Python 桥

子进程 + JSONL：最朴素的跨语言 RPC

一句话：Go 启动 python3 子进程，双方把 JSON 塞到 stdin/stdout，一行一条消息。

对应代码

  • go/internal/worker/client.go — Go 侧发请求、读响应、管生命周期
  • python/bridge/protocol.py — Python 侧序列化 helper
  • python/main.py — Python worker 的读-分发-写主循环

为什么这么写

  • 一行一条 JSON：行分隔就是天然协议边界，不用写长度前缀或 protobuf。
  • select + context：读 worker 响应放到 goroutine 里，主线程用 select 监听 ctx.Done()；超时就杀掉子进程，避免卡死。
  • mu.Lock：保证同一时刻只有一个请求在往子进程写，简化 Python 侧的顺序处理。

背后的工程概念

严格说这是工程部分，不是 AI 概念，但它是本项目"能跑起来"的关键：真实生产里，AI 服务经常是一个独立的 Python 进程/容器，Go 侧做编排和接入层，同样的模式换成 gRPC / HTTP 都行，核心思想是隔离关注点、隔离依赖。

动手练习

  1. 把通信从 stdin/stdout 换成 HTTP：在 Python 侧用 FastAPI 暴露 /call，Go 侧用 http.Client 发请求。比较两种实现的启动时延、错误处理复杂度。
  2. 在 Go 侧加重试：Python 崩了自动重启 worker（需要改 Start 和 Call，捕获 stdin 写失败）。
章节 7 · 本地模型适配

LocalModelAdapter：真模型接入的唯一点

一句话：所有 LLM 调用走 LocalModelAdapter，当前是 mock，以后换真模型只动这一个文件。

对应代码

python/llm/local_model.py

为什么这么写

  • 接口稳定：签名固定是 (prompt, product_info, ...) -> (raw_output, AffiliateCopy)，换真模型只需要保证这个契约。
  • mode 开关：Runtime 通过环境变量 AFFILIATE_MODEL_MODE 决定走 mock 还是真模型（见 bridge/runtime.py L32-L35）。
  • deterministic mock：mock 的输出跟输入有稳定关系，方便写单元测试（见 python/tests/test_prompt_lab.py）。

背后的 AI 概念

模型抽象层是任何要上生产的 AI 系统必备的：方便切换供应商（OpenAI/本地/Claude）、做降级兜底、做测试桩。LocalModelAdapter 就是这个抽象最小实现。

动手练习

  1. 把 mode == "ollama" 分支实现出来：内部用 ollama 的 HTTP API 调本机模型，拿到原始文本后 json.loads 再 AffiliateCopy.model_validate。
  2. 在 adapter 里加一个简单的 LRU 缓存：相同 prompt 直接复用结果，能显著降低调试期的响应延迟。
  3. 进一步：把 generate_structured_copy 的内部实现切成"先让 LLM 自由生成 → 再用 Pydantic 校验 → 失败时降级到规则"。
章节 8 · 实战路径

按这个顺序练，概念就能和代码对上

不推荐一上来就跳到某一章改，按下面的顺序练，你会自然地把整条 AI 编排的肌肉记起来。

  1. 第 1 步 · 跑通全链路

    打开 Studio → 运行 → 对着 trace 一个个节点看。目标：把第 0 章的时序图亲手走一遍。

  2. 第 2 步 · 改 Prompt 和 Schema

    到第 1、2 章的练习：改一句 system prompt、加一个字段。体会"结构化输出"为啥比让 LLM 自由发挥稳。

  3. 第 3 步 · 加一个 Tool

    第 4 章：新建 inventory_tool，从 fixture 查库存。重点练"tool 失败要抛结构化异常，不是 return 默认值"。

  4. 第 4 步 · 改图：加节点 + 加条件边

    第 5 章：把 inventory_tool 的结果做成新节点接在 check_commission 之后，并加第三个分支 needs_review。

  5. 第 5 步 · 换真 RAG

    第 3 章：把 KeywordRetriever 换成 sentence-transformers + Chroma。接口保持不变是关键。

  6. 第 6 步 · 换真模型

    第 7 章：实现 LocalModelAdapter 的真实分支（Ollama / transformers）。这一步做完你就有了一个完全本地的 affiliate AI。

  7. 第 7 步 · 可选：做真 Agent

    把某个节点换成"让 LLM 自己决定调哪个 tool" —— 就是从 LangGraph 的确定性图，升级到 ReAct / tool-calling agent。
