# Channel Playbooks：分渠道打法

**同一个策略在不同渠道上，措辞、节奏、抄送对象都要变。** 选错渠道 = 好牌打成烂牌。

---

## 1. 渠道权重表

| 渠道 | 留痕强度 | 受众范围 | 响应期望 | 政治风险 |
|------|---------|---------|---------|---------|
| 私聊 IM | 低（对方可删） | 1 人 | 即时 | 低，但证据弱 |
| 群 IM | 中 | 5-50 人 | 小时级 | 中 |
| 项目文档（Confluence / 飞书） | 高 | 相关方 | 天级 | 低（正式感） |
| Jira / Issue 评论 | 高 | 项目相关方 | 天级 | 低 |
| 邮件 | 非常高 | 抄送列表 | 天级 | 中高（"正式"）|
| 1on1 | 低（无证据）| 1 人 | 即时 | 低 |
| 周会 / 部门会 | 中（有纪要） | 整个部门 | 即时 | 高（公开博弈） |
| 季度 review | 非常高 | 大老板在场 | 一次性 | 非常高 |

**原则：** 我方占优的议题→尽量推到高留痕渠道；对方占优的议题→留在低留痕渠道抽身。

## 2. IM（私聊 / 群）

### 私聊 IM

**适用：**

- 试探对方立场
- 收集信息
- 拉盟友的初次接触
- 发泄情绪的草稿（但**不要真发**）

**不要在私聊做的事：**

- 做出重要承诺（对方会截图，你没有相同对等的保护）
- 接任务（容易被说"你明明答应了"）
- 讨论敏感议题（离职、跳槽、HR、对其他同事的评价）

**话术模板：**

> 私聊的定位是"聊一聊"、"对一下"，不是"决定"、"承诺"、"对齐"。重要的结论一律补一句"那我在 <群 / 文档 / 邮件> 里同步一下"。

### 群 IM

**适用：**

- 公开化（把私聊结论搬到群里）
- 对齐多方
- 当众 @ 相关人催进度（合规 @ ，非情绪化）

**节奏技巧：**

- **事实 + 动作**，不评价人："刚才对齐的方案是 X，我会先做 A，<对方> 这边麻烦 <具体动作>，谢谢。"
- **用 @ 而非点名**：@ 比文字点名更合规
- **发完不追问**：发完之后的空气越安静对方压力越大；对方不回复你可以过 24h 补一句"@ X 这个还需要你回一下"

**话术模板（公开化）：**

> @ALL 同步一下 <项目> 当前决议：
> 1. A 已确认，owner：我，完成时间：<日期>
> 2. B 需要 @QA_Gatekeeper 这边协助，预计 <时间> 给到
> 3. 如有不同意见请回复，无回复默认按以上推进。

**避免：**

- 群里发长情绪（失分）
- 群里当众撕（失分，对方反而赢同情）
- 在老板的老板在的群里做不成熟的讨论（老板会觉得你越级）

## 3. 项目文档 / Confluence / 飞书文档

**适用：**

- 沉淀决策
- 留规则 / 流程 / RACI
- 追踪进度

**技巧：**

- **每个议题一个页面** / 一个章节，便于后续引用
- **"决定"和"讨论"分开记**：决定不可改，讨论可加评论
- **修改历史可见**：写完第一版就 @ 相关方 review，对方在评论里留下的每条都是证据

**典型结构：**

```
# <项目 / 议题>

## 背景
（中性描述，不评价人）

## 决议（<日期> <会议>）
1. 决议 1：动作 / owner / deadline
2. 决议 2：动作 / owner / deadline

## 讨论中的议题
- 议题 A：xxx（@ 张三跟进）
- 议题 B：xxx（@ 李四跟进）

## 后续
- <日期> 做 review
```

**钩子埋法：**

- 把对方的模糊表态写成具体条目，在后面 @ 他："这是我理解的结论，对吗？"
- 对方同意 = 书面承诺
- 对方不回 = 默认同意（**只要给了合理等待时间**，如 3 个工作日）

## 4. Jira / Issue 评论（尤其是你现在这个 bug 分配场景）

**适用：**

- 处理具体 bug / ticket 时的流程争议
- 技术分工争议
- 需要绑定具体项目号留证据时

**技巧：**

- **每条评论结构化**：当前状态 / 阻塞原因 / 需要谁做什么
- **@ 到具体人而非组**：组别 @ 容易石沉大海
- **关键节点截图归档**：Jira 评论虽然留痕，但自己备份一份到个人文档（万一日后权限被撤）

**话术模板（bug 转让争议）：**

> 当前状态：bug 已分配到 QA Team，按现有流程需要由 QA 决定具体承接人。
> 因本 ticket 紧急程度为 <级别>，建议直接按 <机器人分配规则 / 文档> 分到具体同学，避免中间环节延迟。
> @ <QA leader>：是否可以直接 assign 到 <执行同学> 处理？

**节奏：**

- 24 小时没回应 → 再 @ 一次 + 抄送 @ Boss
- 48 小时没回应 → 升级到群聊 / 邮件 + 抄送 BigBoss

## 5. 邮件

**适用：**

- 正式决策确认
- 跨部门 / 跨级沟通
- 需要给 HR / 审计留证据时

**节奏技巧：**

- 邮件 = 非常高的正式度。**不要为小事发邮件**（显得你在搞事情）
- 抄送列表就是政治信号。**抄送谁不抄送谁都是表态**
- 邮件**不要用情绪词**，只用事实 + 动作 + 时间点
- 邮件标题要能被搜索到：`[项目名] <议题> - 确认 & 下一步`

**结构模板：**

```
收件人：<决策人>
抄送：<相关方，以及需要让对方"感受到压力"的人>
标题：[项目名] <议题> - 确认 & 下一步

Hi <决策人>,

同步一下 <项目 / 议题> 现状：

1. 背景：<两句话>
2. 当前卡点：<具体问题>
3. 建议方案：<A / B 二选一>
4. 需要的决策：<一句话说清楚你要对方做什么>
5. 时间点：希望在 <日期> 前有个结论

相关链接：<doc / ticket>

谢谢
<你的名字>
```

**抄送策略：**

- 我方盟友 + 对方 leader + （如需加压）对方 leader 的 leader
- **不抄送**：HR（除非是正式申诉）、跟议题无关的其他团队
- **抄送对方的 leader** 本身就是轻度升级信号，对方会在意

## 6. 1on1（与 Boss / BigBoss）

**适用：**

- 关系沟通（不该在群里聊的事）
- 试探立场
- 做向上管理

**核心原则：**

- **1on1 没留痕**。会上的所有承诺都要**会后补一条 IM 总结**（公开化钩子）
- **不谈别的同事坏话**（你以为你在同盟，其实对方可能回头说给当事人听）
- **提前准备议题**（纸质 / 文档），不要被对方主导整个 1on1 的话题

**与 Boss 的 1on1 议题模板：**

```
1. 业务进展（占 40%）
   - A 已完成
   - B 在跑，预计 <时间> 出结果
   - C 卡在 <具体原因>，希望你拍一下

2. 具体问题（占 40%）
   - <你真正想谈的那个事，但包装成业务问题，不要包装成"我对谁不满"}

3. 个人发展（占 20%）
   - <OKR 对齐 / 成长方向>
```

**会后必做：**

> @Boss 谢谢刚才的交流。整理一下关键决定：
> 1. A 的优先级按你说的调整到最高
> 2. C 的卡点你会帮我去跟 <对方> 推一下，本周内给我反馈
> （如有理解偏差请指出）

**这段 @ 就是会后钩子，让口头承诺变书面。**

### 与 BigBoss（Skip）的 1on1

**适用：** 议题升级、了解大方向、建直接通道

**禁忌：**

- **不谈 Boss 的坏话**（哪怕你很想谈）
- **不跳过 Boss 做决策**（你要的是"让 BigBoss 知道"，不是"让 BigBoss 替 Boss 做主"）
- **只聊业务，不聊关系**

**议题模板：**

- 业务现状 + 建议方向
- 遇到的挑战（包装成业务问题）
- 想请教的方向

**如果 BigBoss 主动问你 Boss 的表现：**

> 技巧性回答："Boss 给了我很多支持。我现在想重点对齐的是 <业务问题>，如果这个能解决，我这边的节奏会顺很多。"
>
> —— **把"关系问题"翻译成"业务问题"**，让 BigBoss 从业务视角帮你解决 Boss 的问题。

## 7. 周会 / 部门会

**适用：**

- 公开确认进展
- 让其他人看到我方贡献
- 被当众攻击时的临场应对

**临场应对（被当众攻击）三原则：**

1. **不辩解**："我听到了，这个我会后给一个完整的回复。"
2. **不反击**：当众硬怼 = 显得不成熟，哪怕赢了也输了
3. **留尾巴**：会后立刻发一条 IM 到对方 / 群里："刚才会上提到的 X，我整理了事实经过（附链接），如有疑问欢迎讨论。"

**会后动作（24 小时内）：**

- 发书面回复到群 / 文档（**在你准备好的时候**，不在对方要你回的时候）
- 如必要，约 1on1 私下沟通
- 保存会议纪要截图

## 8. 季度 Review / 绩效沟通

**适用：**

- 总结贡献
- 谈绩效 / 评级 / 晋升
- 反制"秋后算账"

**关键原则：**

- **全年的事实要提前准备** —— 做一份"项目 + 数据 + 证据链接"的个人 brag doc，review 时随时能引
- **不抱怨、不比较**（不说"别人做得少还评得高"）
- **争议点书面化**：如果对 Boss 的评价有异议，**要求在 review 记录里加一段你的补充说明**

**话术模板（被打低评时）：**

> 感谢反馈。我这边对评级理解是 <对方给的评级>。关于你提到的 <具体问题>，我整理了对应的事实和数据（附文档链接），希望记录在系统里，便于后续对齐。

## 9. HR 渠道

**适用：**

- 正式申诉（骚扰、歧视、非法解雇、报复）
- 需要第三方调查的事件

**极度谨慎原则：**

- **只去一次**（HR 的定位是保护公司，不是保护你；去之前要做好换组 / 离职的准备）
- **书面 > 口头**（HR 记录是公司记录，口头的会失真）
- **证据完整**（IM 截图、邮件、日期、在场人）
- **找到对应的 HRBP 而不是随便一个 HR**

**邮件模板（首次申诉）：**

```
标题：[正式反馈] 关于 <简短描述>

Hi <HRBP>,

我希望正式记录并讨论以下事件：
1. 事件描述（时间、地点、在场人、经过）
2. 附件：相关证据（截图、邮件）
3. 已尝试的沟通：<你尝试过的动作>
4. 希望的处理方式：<调查 / 对齐 / 调岗>

方便的话本周安排一次沟通。
```

**提交前务必：**

- 自己先想清楚"希望的处理方式"，不然 HR 会把问题踢回来
- 确认你和 Boss 的关系已经无法修复（或者你不在乎修复）
- 做好换组 / 离职的准备

## 10. 选渠道速判表

| 当前场景 | 首选渠道 | 备选渠道 |
|---------|--------|--------|
| 试探对方立场 | 私聊 IM | 1on1 |
| 埋书面化钩子 | 群 IM / 文档 | 邮件 |
| 推进具体 bug / 项目 | Jira 评论 | 群 IM |
| 跟 Boss 对齐决策 | 1on1 + 会后 IM 总结 | 邮件 |
| 跨部门推诿 | 邮件（抄送多方） | 群 + @ 双方 leader |
| 被当众攻击 | 当场不接招，会后文档回复 | 1on1 私下沟通 |
| 秋后算账 / 绩效 | 季度 review + 书面记录 | 1on1 + HR |
| 真正撕破脸 | HR 邮件 | Skip 1on1 |

---

## 渠道原则速记

1. **我强推书面化，对方弱推去正式**
2. **抄送策略 = 政治策略**
3. **重要承诺永远会后补一条 IM**
4. **当场不回应，会后才回复**（你掌握节奏 = 你掌握主动权）
5. **自己的个人设备备份一切关键证据**
