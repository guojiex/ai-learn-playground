# Case: Bug 分配博弈（QA_Gatekeeper + Boss 组合拳）

这是本 skill 的第一个示范案例，完整走一遍「输入 → 局势判断 → 三档输出 → 钩子 → 反击」的闭环。**所有人名、公司名已脱敏**，角色代称统一使用。

---

## 输入（用户提供）

```
【我是谁】Me，某业务线研发
【对方是谁】
  - Boss：我的直属老板，有 PUA 倾向，擅长用 QA_Gatekeeper 做白手套
  - QA_Gatekeeper：QA Leader，擅长推三阻四扩大影响面
【渠道】飞书 / 钉钉 IM（截图是群聊 / 半群聊上下文）
【原始对话】
  Me：建议把 bug 直接 assign 到具体 QA 同学，而不是先 assign 给 QA Leader 再转手。
  QA_Gatekeeper：需要一个文档 / 或者做个 bot 才行。（推三阻四）
  Me：做个 bot 吧，直接 assign 给 bot 就好了。
  QA_Gatekeeper：<AI 战略代号> 不是说要支持 AI 能力吗，正好这就有业务场景了，很棒。
  QA_Gatekeeper：我们没有人力支持这个。
  Me：看看开发做一下吧。
  QA_Gatekeeper：我们有很多 ai 项目在进行中。
  Boss（介入）：这个事情，对你们来说成本就是一次转让时指定人，是 QA_Gatekeeper 还是
  按照机器人分配的 qa，成本是一样的，至于准确与否是机器人分拣要保证的事情。至于为
  什么不建议都给 QA_Gatekeeper，因为这样她得去分配，多了一个中心点。万一感知处理晚
  了或者休假，bug 的处理就会不及时。
【我的诉求】
  1. 不让 QA_Gatekeeper 成为所有 bug 的单点入口（她借机扩大影响、推三阻四）
  2. 让 Boss 不能再用"成本一样""中心点风险"这种话术堵我
  3. 让议题公开化，必要时升级
  4. 不撕破脸，保留后续合作空间
【红线】
  - 不想直接跟 Boss 或 QA_Gatekeeper 撕破脸（还要长期共事）
  - 不想被贴"不配合""情绪化"标签
```

---

## 1. 局势判断

典型的**白手套组合拳**：QA_Gatekeeper 挡前面推三阻四扩大影响；Boss 用"成本模糊化 + 单点风险 + 休假共情"四板斧帮白手套压我。目标是把流程卡在 QA leader 手里，让他们有更大的流程话语权。

---

## 2. 对方话术解构

对照 [../pua-counters.md](../pua-counters.md)：

| 对方原话 | 话术类型 | 真实意图 |
|---------|---------|---------|
| "需要文档 / 做个 bot 才行" | 拖延（[tactics-defense.md](../tactics-defense.md) 对手版的"流程化"）| 把议题无限延后 |
| "<AI 战略代号> 不是说要支持 AI 吗" | C3 转述式挡箭（虚指一个权威）| 用大词转移议题 |
| "我们没有人力支持" | A3 人力挡箭 | 不可验证 |
| "有很多 AI 项目在进行中" | A3 延续 | 强化挡箭 |
| "成本一样" | **A1 模糊化** | 抹平你的真实成本 |
| "机器人分拣要保证准确" | **C1 甩系统** | 把责任推给没有主体的"系统" |
| "多了一个中心点" | **A2 中心点反用** | 把"有负责人"定义为"风险" |
| "万一休假" | A2 共情包装 | 让反对方案显得是在"关心 QA_Gatekeeper" |

**组合识别：典型的 E1（复合诡辩）。单点反驳任何一句都会被另外几句压回来。**

---

## 3. 我方筹码与风险

**筹码（盘一盘）：**

- 有实际的业务数据（bug 处理时延、QA leader 转让的 lag）
- QA_Gatekeeper 的"没人力"声明是模糊的，数字一拉就破
- 产品 / 下游可能有类似痛点，可以拉盟友
- "单点 / 休假"风险本身可以反用到 QA_Gatekeeper 身上

**风险：**

- Boss 已亲自下场，直接硬怼 Boss 风险较高
- QA_Gatekeeper 借"推三阻四"已积累一些"流程合规"形象，不好直接否掉
- 私聊 IM 渠道留痕弱

**风险六项打分（[framework.md](../framework.md) 第 4 节）：**

| 维度 | 评估 |
|------|-----|
| 我离职成本 | 中（暂不考虑立即走） |
| Boss 报复能力 | 中高 |
| 事件可公开度 | 中（目前私群，可推到项目文档 / Jira）|
| 我方数据 | 较充分 |
| 盟友数量 | 中（产品可能站我） |
| 业务重要性 | 中 |

**档位初选：标准 → 偏温和，先推书面化 + 数据化，避免直接硬怼 Boss。**

---

## 4. 三档方案

### 温和档

**回复原文（回给 QA_Gatekeeper，不直接回 Boss）：**

> @QA_Gatekeeper 收到，这块我理解大家也都忙。为了让我这边也能给产品一个说法，能帮忙拉齐几个数字吗：
>
> 1. QA 侧目前 bug 转让的平均延迟是多少？
> 2. 如果要做 bot，大概需要多少人日？对应是哪位同学？
> 3. 这季度 QA 的 AI 项目 owner 是哪几位？我好同步一下产品预期。
>
> 我这边先整理一份 bug 延迟的数据，稍后同步到 <项目文档>。

**埋下的钩子：**

- **钩子 1：成本量化钩子**（[../hook-patterns.md](../hook-patterns.md) §2）—— "需要多少人日？对应哪位同学？"
  - 触发信号：对方给不出数字 / 给出夸张数字
- **钩子 2：责任归位钩子**（§3）—— "AI 项目 owner 是哪几位？"
  - 触发信号：对方含糊其辞，说明"AI 项目在进行中"是借口
- **钩子 3：时间戳钩子**（§8，隐形版）—— "我这边先整理数据同步到文档"
  - 触发信号：对方若不回，你公开化动作已生效

**预期对方走向：**

- 40%：QA_Gatekeeper 拖延 / 不回 → 你走公开化，胜
- 30%：给出夸张人日 → 你反问拆分，胜
- 20%：老板再次介入帮 QA 说话 → 进入下一轮反击
- 10%：妥协同意 → 直接赢

---

### 标准档

**回复原文（在群里 / 项目群里发，@Boss 和 @QA_Gatekeeper）：**

> @Boss @QA_Gatekeeper 基于刚才的讨论，我这边整理一下现状和下一步建议：
>
> **现状：**
>
> 1. bug 目前先 assign 给 QA leader 再转让，有 N 小时的平均延迟（我会同步详细数据）
> 2. 关于"机器人分拣准确性"的顾虑，当前准确率需要对齐一个数据
> 3. 关于"leader 是单点"的顾虑，标准做法是 primary + backup 而非"不设 owner"
>
> **建议：**
>
> 1. 先把 bug 分配延迟、分拣准确率这两个数字拉齐 —— 一周内出
> 2. 基于数据决定方案：是改进机器人（owner：@？）还是改进转让 SOP（owner：@QA_Gatekeeper）
> 3. 无论哪个方案，都需要明确 primary + backup 的 SOP
>
> 如无异议，我这周内先出数据 + doc 草案，大家 review。

**埋下的钩子：**

- **钩子 1：书面化钩子**（§1）—— "整理 doc 同步，如无异议我推进"
  - 触发信号：对方若反对公开化 → 强钩
- **钩子 2：责任归位钩子**（§3）—— "owner 是 @？"
  - 触发信号：对方不愿意明确 owner → 问题显形
- **钩子 3：时间戳钩子**（§8）—— "一周内出数据"
  - 触发信号：对方若给不了 deadline，你自己定
- **钩子 4：优先级确认钩子**（§7，隐形版）—— "是改机器人还是改 SOP"
  - 触发信号：对方不拍板，你拉 Boss

**预期对方走向：**

- 50%：Boss 不方便公开反对（因为你没攻击他），QA_Gatekeeper 需要配合
- 30%：QA_Gatekeeper 试图拖延出数据 → 你有 deadline 可以追
- 20%：Boss 再次亲自介入，但议题已在群里公开，他无法重复原来的模糊话术

---

### 强硬档

**回复原文（项目 doc + 邮件，抄送 BigBoss 或者同级产品 leader）：**

> Subject: [QA 流程] bug 分配 SOP 对齐 & 数据 review
>
> Hi @Boss @QA_Gatekeeper，抄送 @产品 Leader，
>
> 关于 bug 分配流程，同步一下当前情况和建议，请帮忙决策：
>
> **背景：**
>
> 当前 bug 先 assign 到 QA leader 再由 leader 人工转让，对处理时延和下游体验有影响。我们讨论了「直接由机器人 assign 到具体 QA」的方案。讨论中主要有以下几个顾虑：
>
> 1. 成本差异
> 2. 机器人分拣准确率
> 3. Leader 成为"中心点"的风险
>
> **为支持决策，我建议对齐以下数据：**
>
> 1. 当前 bug 转让平均时延（owner：我，<日期>前）
> 2. 当前机器人分拣准确率（owner：@机器人系统 owner，<日期>前）
> 3. 成本拆分对比（时间 / 响应 / 累计）
>
> **两个待决策项：**
>
> 1. 是否按 primary + backup 机制设计 SOP？
> 2. 机器人分拣改进的 owner 是谁（QA 侧 or 工具侧）？
>
> 建议 <日期> 前对齐。如需拉会我可以约。
>
> Thanks

**埋下的钩子：**

- **钩子 1：公开化钩子**（§5）—— 抄送产品 leader + doc
  - 触发信号：对方若私下跟你说"别发这么正式的邮件" → 强钩
- **钩子 2：责任归位钩子**（§3）—— "机器人系统 owner"
  - 触发信号：无法 / 不愿意指出 owner
- **钩子 3：时间戳钩子**（§8）
- **钩子 4：文档引用钩子**（§9）—— 把"primary + backup" 作为标准做法暗指对方反对标准做法没理

**预期对方走向：**

- 30%：Boss 会找你 1on1 批评"你为什么发这么正式的邮件" → 正好触发公开化钩子的反击，记录
- 30%：QA_Gatekeeper 被迫给数据或 owner
- 20%：BigBoss 介入拍板 —— 多数情况下会让对等数据对齐，对你有利
- 20%：产品 leader 站出来说"我们也有这个痛点" —— 盟友形成

**代价：** Boss 会明确感受到你在"搞事情"。**仅在你已经做好换组 / 换工作预案时再用强硬档**。

---

## 5. 推荐档位

**推荐：标准档**。理由：

1. 你的红线是"不撕破脸、保留合作空间"，强硬档成本太高
2. 温和档无法有效回应 Boss 那句"四板斧"，Boss 会继续压
3. 标准档在群里公开、@ 双方，措辞中性，把议题推向数字化 + 书面化 —— 对方无法反对"拉数字"
4. 如果标准档后对方继续耍赖，**再升级到强硬档会更有理**（"我已经尝试过温和对齐"是很好的升级理由）

## 6. 下一步监测 & 升级路径

### 48 小时内监测信号

| 对方反应 | 解读 | 下一步 |
|---------|-----|-------|
| QA_Gatekeeper 给出数据 | 议题进入理性轨道，继续推 | 按数据讨论方案 |
| QA_Gatekeeper 拖延不给数据 | 触发量化钩子，继续追 | 48h 后 @ 一次 + 抄 Boss |
| Boss 帮 QA 说话，用新话术 | 进入反击模式 | 进入第二轮，读本案例下方 |
| Boss 找你 1on1 说"你别搞这么正式" | 触发公开化钩子 | 1on1 后发 IM 总结"谢谢 1on1 提醒，数据对齐方向我们还是按原计划推" —— 让他无法私下撤销你的进展 |
| 无任何回应 | 沉默即默认 | 3 个工作日后自行推进 doc，同步到群 |

### 升级路径

- 3 天内没有数据对齐 → 软升级：周报里写上这个阻塞点，同步到更大群
- 议题超过 2 周无进展 → 议题升级：约 BigBoss 1on1 谈业务（不谈人）
- 对方出现明显打压 / 报复行为 → 关系升级准备：做好证据备份 + 确认换组 / 离职预案

## 7. 模拟第二轮（假设对方回复）

这部分演示 **反击模式（Counter Mode）** 怎么工作。

### 假设的对方回复（Boss 接着发）：

> "不用这么正式吧，这个事情本来就是流程小问题，你们不要小题大做。QA_Gatekeeper 已经在做了，你们配合一下就行。"

### 钩子回收表

| 钩子 | 是否触发 | 对方原话 | 强度 |
|------|---------|---------|-----|
| 公开化钩子（标准档 #1）| **强触发** | "不用这么正式" = 反对书面化 | 强 |
| 责任归位钩子（#3）| **绕开** | 没说 owner 是谁 | 中 |
| 时间戳钩子（#4）| **绕开** | 没给时间 | 中 |
| 优先级确认钩子（#5，隐形）| **绕开** | 没拍板 | 中 |

**整体动作识别：** Boss 用"小题大做 / 正式 / 配合"三连来压你。**情绪施压 + 人设 PUA + 指责你不配合**。

### 第二轮三档反击

#### 温和档

> @Boss 理解你希望轻量。我这边会先跟 QA_Gatekeeper 对一下 bug 转让时延的数据，基于数据再看要不要改流程。如果数据没问题，我们就按现状走。本周我会同步一下结果。

**埋新钩子：** 时间戳钩子 + "基于数据"隐形钩子 —— 把对方逼到数据面前。

#### 标准档

> @Boss 我也希望轻量推进。同步一下我的思路：
>
> - 这件事本身确实小，但**没有数据 / 没有 SOP** 的情况下，每次遇到都会重复讨论，长期 cost 反而更高。
> - 所以我只想先做一件事：**把 bug 转让的现状数据拉一下**。如果数据显示现状没问题，我们就维持现状，我也就不再提；如果有问题，再讨论方案。
> - 数据我本周内出，不需要 QA 配合任何开发动作。这样轻量吗？

**埋新钩子：**

- **一致性挑战钩子**（[../tactics-defense.md](../tactics-defense.md) §6）—— 把对方"流程小问题"的话还给他："正因为小问题，一次对齐数据反而最省事"
- **优先级确认钩子**：逼对方在"拉数据 / 不拉数据"里选一个
- **互惠陷阱钩子**：我让步（数据自己出），对方难以继续阻拦

#### 强硬档

> @Boss 谢谢提醒。我想和你对齐一下我的理解：
>
> - "不用正式" —— 我把现状 + 建议整理成 doc 是为了减少后续反复讨论，不是把事情做大
> - "QA_Gatekeeper 已经在做了" —— 我没有看到具体动作同步。能不能请 @QA_Gatekeeper 同步一下当前进展和 deadline？这样我也能对齐给产品
> - "你们配合一下" —— 我们一直在配合。建议把"配合"的具体定义也对齐一下，避免理解偏差
>
> 这些要求我觉得都不过分，能帮忙回应下吗？

**埋新钩子：**

- **衡量标准钩子**（[../hook-patterns.md](../hook-patterns.md) §6）—— 把"配合"这个模糊 PUA 词逼到具体
- **责任归位钩子 V2** —— 把 QA_Gatekeeper 的进度要求摊在桌面
- **一致性挑战钩子** —— 把"不用正式" "小题大做"等 PUA 话术合规化反问

**代价：** Boss 会觉得你在抗。**只在前面温和 / 标准档已做过、你已积累证据时才升到这一档**。

### 第二轮推荐

**推荐标准档**。理由：

- 用"我让步"（数据我自己出）消解了"不配合"指控
- 把议题继续钉在"拉数据"这个对方无法拒绝的点上
- 对方如果再反对"拉数据"，就是明显的理亏

## 8. 从这个案例要记住什么

1. **复合诡辩（E1）不要一条一条反驳**，要换框（从"方案细节"换到"责任归属 + 数据"）
2. **钩子比表态重要**：每一轮都要埋新钩子
3. **让步要让在无关紧要的点上**（数据我自己出），大头不退
4. **渠道决定胜负**：从私聊 → 群 → 文档 → 邮件，逐步推高书面化层级
5. **不要硬怼 Boss 本人**，只怼议题、只回具体事实
6. **PUA 话术（"小题大做""配合""不用正式"）要识别出来再回**，不要接它们的词
